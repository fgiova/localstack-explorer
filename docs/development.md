# Development Guide

This document covers how to set up, develop, test, and contribute to LocalStack Explorer.

## Prerequisites

- **Node.js** >= 20
- **pnpm** >= 9 — install with `npm install -g pnpm` or [see pnpm docs](https://pnpm.io/installation)
- **Docker** — for running LocalStack

### Starting LocalStack

A `docker-compose.yaml` is provided at the project root with all required services pre-configured:

```bash
docker compose up -d
```

This starts LocalStack with S3, SQS, SNS, IAM, CloudFront, and CloudFormation on `http://localhost:4566`.

To stop it:

```bash
docker compose down
```

## Setup

```bash
# Clone and install
git clone <repository-url>
cd localstack-explorer
pnpm install
```

## Development Workflow

### Running the Dev Servers

```bash
# Both backend and frontend in parallel
pnpm dev

# Or separately:
cd packages/backend && pnpm dev    # http://localhost:3001
cd packages/frontend && pnpm dev   # http://localhost:5173
```

The frontend Vite dev server proxies all `/api/*` requests to the backend at `localhost:3001`, so you can access the full application at `http://localhost:5173`.

### Environment Variables

Create a `.env` file in `packages/backend/` (or the project root) if you need non-default settings:

```bash
PORT=3001
LOCALSTACK_ENDPOINT=http://localhost:4566
LOCALSTACK_REGION=us-east-1
ENABLED_SERVICES=s3,sqs,sns,iam,cloudformation
```

The `ENABLED_SERVICES` variable accepts a comma-separated list of service names. Only the listed services will have their backend routes registered and their UI visible in the dashboard and sidebar. Available values: `s3`, `sqs`, `sns`, `iam`, `cloudfront`, `cloudformation`.

### Route Generation

TanStack Router uses file-based routing. When you add, rename, or delete a file in `packages/frontend/src/routes/`, the route tree is automatically regenerated by the Vite plugin. If you need to regenerate manually:

```bash
cd packages/frontend
npx @tanstack/router-cli generate
```

## Testing

### Running Tests

```bash
# All tests
pnpm test

# Backend only
cd packages/backend && pnpm test

# Frontend only
cd packages/frontend && pnpm test

# Watch mode (useful during development)
cd packages/backend && npx vitest
cd packages/frontend && npx vitest
```

### Backend Test Approach

Backend tests use `fastify.inject()` to simulate HTTP requests without starting a real server. Services are mocked to isolate route-level behavior.

Test files live in `packages/backend/test/` mirroring the `src/` structure.

Example test structure:

```typescript
import { describe, it, expect, vi, beforeAll, afterAll } from "vitest";
import Fastify, { FastifyInstance } from "fastify";
import { registerErrorHandler } from "../../src/shared/errors.js";
import { s3Routes } from "../../src/plugins/s3/routes.js";

describe("S3 Routes", () => {
  let app: FastifyInstance;

  beforeAll(async () => {
    app = Fastify();
    registerErrorHandler(app);
    const mockService = { /* mock methods with vi.fn() */ };
    await app.register(s3Routes, { s3Service: mockService });
    await app.ready();
  });

  afterAll(() => app.close());

  it("GET / returns buckets", async () => {
    const res = await app.inject({ method: "GET", url: "/" });
    expect(res.statusCode).toBe(200);
  });
});
```

### Frontend Test Approach

Frontend tests use React Testing Library to render components and assert on DOM output. API hooks are mocked with `vi.mock()`.

Test files live in `packages/frontend/test/` mirroring the `src/` structure.

Components that depend on TanStack Router must be wrapped with a `RouterProvider` and `QueryClientProvider`. Use `waitFor()` for assertions that depend on async rendering.

## TypeScript

### Type Checking

```bash
# Backend
cd packages/backend && npx tsc --noEmit

# Frontend
cd packages/frontend && npx tsc --noEmit
```

### Configuration

- `tsconfig.base.json` (root) — shared strict settings
- `packages/backend/tsconfig.json` — extends base, outputs to `dist/`
- `packages/frontend/tsconfig.json` — extends base, `noEmit` (Vite handles bundling), `@/*` path alias

### Path Aliases

Both packages support the `@/*` alias:

- Backend: not configured (uses relative imports with `.js` extensions for ESM)
- Frontend: `@/*` maps to `./src/*`, configured in both `tsconfig.json` and `vite.config.ts`

### ESM Imports

The backend uses ES modules. All local imports must include the `.js` extension:

```typescript
// Correct
import { config } from "./config.js";

// Incorrect — will fail at runtime
import { config } from "./config";
```

## Linting & Formatting

The project uses [Biome](https://biomejs.dev/) as a unified linter and formatter. The configuration is defined in `biome.json` at the repository root and applies to all `.ts` and `.tsx` files across both packages.

### Running Checks

```bash
# Full check (lint + formatting + import sorting)
pnpm check

# Auto-fix all fixable issues
pnpm check:fix

# Lint only
pnpm lint
pnpm lint:fix

# Format only
pnpm format
pnpm format:fix
```

### Configuration Overview

| Setting | Value |
|---|---|
| Indent style | Tabs |
| Quote style | Double quotes |
| Lint rules | Recommended |
| Import sorting | Automatic (`organizeImports`) |
| VCS integration | Git (`.gitignore` respected) |

### Excluded Files

The following paths are excluded from analysis:

- `node_modules`, `dist`, `build` — build artifacts
- `**/routeTree.gen.ts` — auto-generated by TanStack Router

### Rule Overrides

| Rule | Level | Reason |
|---|---|---|
| `suspicious/noArrayIndexKey` | `warn` | Allowed in dynamic form fields without stable IDs |
| `a11y/noLabelWithoutControl` | `off` | Generic `Label` component receives `htmlFor` via props spread |

### Editor Integration

Install the [Biome extension](https://biomejs.dev/guides/editors/first-party-extensions/) for VS Code or JetBrains to get real-time diagnostics and format-on-save.

## Coding Standards

### TypeScript

- **No `any`** — use specific types, generics, or `unknown` with type guards.
- **TypeBox schemas** — define all request/response shapes in `schemas.ts` files. Derive TypeScript types with `Static<typeof Schema>`.
- **Functional React** — only function components with hooks. No class components.

### Backend

- **Plugin isolation** — each service is a self-contained Fastify plugin.
- **Service layer** — business logic goes in `service.ts`, not in route handlers.
- **Error mapping** — catch AWS SDK errors and throw `AppError` with appropriate HTTP status codes.
- **Validation on everything** — every route must have TypeBox schemas for all inputs (params, query, body) and outputs (response).

### Frontend

- **Colocation** — API hooks in `src/api/`, components in `src/components/<service>/`, routes in `src/routes/<service>/`.
- **Query invalidation** — mutations must invalidate relevant query keys so the UI stays in sync.
- **Shadcn/ui components** — use primitives from `src/components/ui/`. Add new ones from the [Shadcn/ui registry](https://ui.shadcn.com/) as needed.

## Building for Production

```bash
pnpm build
```

This runs `tsc` in the backend (compiling to `packages/backend/dist/`) and `tsc + vite build` in the frontend (outputting to `packages/frontend/dist/`).

To serve the production build:

```bash
cd packages/backend && pnpm start
```

In production, you would serve the frontend static files via a reverse proxy (e.g., nginx) or configure the Fastify backend to serve them with `@fastify/static`.
